# Char limit benchmark (TEST ONLY)

사이드패널에서 탭 본문 추출 → **5200 / 8000 / 12000** 각각 요약 → 로컬 JSON 저장.

> 프로덕션 기본값은 **`MAX_INPUT_CHARS = 8000`** (`src/geminiClient.ts`).  
> 벤치마크 모드는 `src/benchmarkConfig.ts`의 `BENCHMARK_MODE`로 켜고 끕니다. **main 기준 `false`.**

---

## 사용

1. `benchmarkConfig.ts`에서 `BENCHMARK_MODE = true`
2. `yarn build` → Chrome `dist/` 리로드
3. 기사 탭 → **Run char-limit benchmark (TEST)**
4. 상단 패널에서 확인·복사 / Export JSON

## 저장 (chrome.storage.local)

키: `benchmarkRunLog` (최대 50건)

각 기록 포함:

- `url`, `language`, `savedAt`
- `extracted`: `pageTitle`, `sentenceCount`, `originalChars`, `bodyText` (원문)
- `runs[]`: 한도별 `stats` + `summary` (read/skip, 3줄, fullSummary)

## 패널 버튼

| 버튼 | 동작 |
|------|------|
| **Copy this run** | 방금 1건 JSON → 클립보드 |
| **Copy all** | 누적 전체 JSON |
| **Export JSON** | 파일 다운로드 |
| **Clear log** | storage 초기화 |

## 설정 파일

- `src/benchmarkConfig.ts` — `BENCHMARK_MODE`, `BENCHMARK_CHAR_LIMITS`
- `src/benchmarkRunLog.ts` — storage

## 검사 결과와 결정 (2026-06-18)

BBC·Wired 등 **10개 URL**에서 본문 추출 후 **5200 / 8000 / 12000** 한도로 각각 요약해 비교했다.  
모델은 전건 `gemini-2.5-flash-lite`, UI 언어 English.

### 관찰

**1. 일반 뉴스(추출 2k~8k)가 대부분이었다.**

Readability로 뽑은 BBC 기사 대부분이 8,000자 미만이었다. 이 구간에서는 8000·12000 한도 모두 **같은 본문**이 모델에 전달되었다.

**2. 5200은 뉴스에서 본문 앞부분만 잘렸다.**

원문이 5k~7k인 기사에서 5200은 tail이 잘리고, 후반 팩트가 요약에서 빠지는 경우가 있었다. 원문 자체가 5200 이하인 페이지는 세 한도 모두 동일했다.

**3. 8000과 12000은 뉴스에서 요약 품질 차이가 거의 없었다.**

원문 &lt; 8k인 8건 모두 입력 본문이 동일했고, fullSummary 길이·내용 차이는 실행마다 달라지는 수준이었다. 12000이 항상 더 낫지는 않았다.

**4. 12000은 장문(2만 자 이상)에서만 체감 차이가 났다.**

Wired·Experimental History처럼 원문이 8k를 넘는 글에서는 12000이 더 많은 앞부분을 읽어 요약이 더 풍부해졌다. 다만 12k 한도로도 원문 전체의 절반 정도만 전달되며, 후반은 여전히 잘렸다.

**5. read/skip 판별은 이번 샘플로 검증되지 않았다.**

30건 모두 `readRecommendation: read`였다. skip이 나와야 할 페이지는 이번 범위에 없었다.

### 결정

| 항목 | 결정 | 근거 |
|------|------|------|
| **`MAX_INPUT_CHARS`** | **8000 유지** | 뉴스 탭 대상 앱에서 추출 본문 대부분이 8k 미만이고, 8000이면 전문 전달에 충분했다 |
| **5200** | **채택 안 함** | 5k~7k 기사에서 tail 손실·팩트 누락 관찰 |
| **12000** | **프로덕션 기본값으로 올리지 않음** | 뉴스 구간에서 입력·품질 차이 없음. 장문 전용 옵션이 필요해지면 그때 별도 검토 |
| **기본 모델** | **`gemini-2.5-flash-lite` 유지** | 벤치마크 전건 1차 모델로 성공, 폴백 없이 동작 |

### 테스트 범위

- **1차 (6 URL):** BBC 뉴스 3, BBC 비디오 1, Wired 장문, Experimental History 장문
- **2차 (4 URL):** BBC 뉴스 — Gilgo Beach 판결, Iran·관광, 기타 2건

## 원문 스크립트와 글자 한도 (보충, 미벤치)

위 벤치마크는 **영문 페이지 원문**만 다뤘다. 앱은 **UI 언어(English / 中文)와 무관**하게 동일한 `MAX_INPUT_CHARS`로 `slice`한다.  
UI가 中文이어도 BBC 영문 기사를 요약하면 **입력은 영어**이고, 벤치 결과가 그대로 적용된다.

### 관찰 (코드·토큰화 특성, 실측 아님)

**1. 한도는 글자 수이지 토큰 수가 아니다.**

`geminiClient.ts`는 본문을 `slice(0, MAX_INPUT_CHARS)`만 한다. Gemini 청구·quota는 **토큰** 기준이다.

**2. 같은 글자 수라도 스크립트마다 토큰 밀도가 다르다.**

대략적인 감(동일 **8,000자**를 꽉 채웠을 때):

| 원문 스크립트 | 글자당 토큰 (감) | 8,000자일 때 입력 토큰 (감) |
|---------------|------------------|-----------------------------|
| 라틴(영어 등) | ~3.5~4자/토큰 | ~2,000~2,500 |
| 한글 | ~1.5~2.5자/토큰 | ~3,500~5,500 |
| 한자(중국어 등) | ~1~2자/토큰 | ~4,000~7,000 |

**3. 같은 **내용**의 기사라면 글자 수는 중국어 &lt; 한국어 &lt; 영어인 경우가 많다.**

번역본 기준으로는 **토큰 총량**이 글자 수 차이보다 더 비슷해지는 편이다(대략 ±20~30%).  
문제는 **8,000자 상한에 걸릴 때**: 한·중 원문은 영어와 같은 글자 한도로 **더 무거운 요청**이 된다.

**4. UI의 토큰 추정(`chars ÷ 4`)은 영어 편향이다.**

메타 표시용 `approxInputTokensHint`는 전 스크립트에 `length / 4`를 쓴다. 한·중 원문에서는 **실제 `promptTokenCount`보다 과소 추정**된다.

### 결정 (현재)

| 항목 | 결정 | 근거 |
|------|------|------|
| **언어별 char cap** | **도입 안 함 (당분간)** | 제품·벤치 표본이 영문 뉴스 중심; 中文 UI ≠ 中文 원문 |
| **`MAX_INPUT_CHARS = 8000`** | **위 벤치 결론 유지** | 영문 추출 길이 기준으로는 충분 |
| **한·중 원문 페이지** | **별도 벤치 전까지 가정만** | 토큰 공정성을 맞추려면 원문 스크립트 감지 후 **더 낮은 char cap**(대략 한글 4k~5.5k, 한자 3.5k~5k) 또는 **토큰 상한** 검토 가능 — **미구현·미검증** |

### 이후 검사 제안

한·중 **원문** 뉴스 URL 각 3~5건, UI는 해당 출력 언어로, **8000 단일 한도**만으로 `promptTokenCount`·요약 품질을 영문 벤치와 비교하면 스크립트별 cap 필요 여부를 판단할 수 있다.
